Case Study 02 — Featured

Kajuju

Designing, building, and operating a boutique lodge in Kenya, and then building the automation framework that runs it.

Role Owner-operator & automation engineer
Stack Playwright, TypeScript, Node.js, Express, Twilio
Infra Vercel, Railway, GitHub Actions

Overview

In 2025, after building Kajuju with family, a boutique lodge in Kenya, alongside the physical property, I built a production test automation framework using Playwright and TypeScript, to run its booking, availability, and guest-communication systems. I treat the lodge the way I'd treat any system I was responsible for testing in a previous role: with synthetic monitoring, CI, and a healthy suspicion of anything that hasn't been verified.

The Problem

A live hospitality business has to keep several systems in tandem, including a channel manager tracking availability across booking platforms and our website, a booking form guests use directly, and a WhatsApp line where some enquiries arrive in addition to phone call enquiries.

For us, the risk is double bookings, or losing out on potential bookings because of slow or missed response or frustrated staff unable to use the software efficiently while doing their work.

The Solution

I built a small set of connected systems, a booking flow with live availability checks, a suite of automated tests running in CI to catch regressions and double-bookings before guests do, and a WhatsApp bot that responds to enquiries and checks availability on a regular automated schedule. Each piece is tested on its own, and as part of the larger system.

Architecture

Guest-facing pages and the booking flow are deployed as static and serverless functions on Vercel. A channel manager holds the availability data, the booking form checks against it in real time before allowing a submission. A separate Node.js and Express service, deployed on Railway, runs the WhatsApp bot via the Twilio API, independent of the main site's deploy cycle. GitHub Actions ties the two together with scheduled test runs against the live, deployed environment.

Automation

The Playwright suite covers the booking flow end to end, form validation, availability checks, and channel manager API responses, plus synthetic monitoring that runs continuously against the live site. A core focus is double-booking detection across multiple room types: the suite checks for date overlaps room by room, on a schedule, and raises an alert the moment two reservations claim the same night.

The WhatsApp bot runs on its own automated schedule too, checking availability and auto-responding to guest enquiries with current rates and booking information, so a guest messaging at 2am gets an answer.

Engineering Decisions

The most instructive bug in the whole project was a timezone drift issue in the double-booking detector. The original overlap check used JavaScript's new Date() to compare a checkout date against a checkin date. new Date() parses a bare date string as midnight UTC, but the property operates several hours ahead of UTC, so a checkout and a checkin on the same calendar day appeared to overlap by several hours. The result was a run of false-positive double-booking alerts that only surfaced in cross-timezone CI runs, not in local testing.

The fix was to stop comparing Date objects and start comparing calendar dates as plain YYYY-MM-DD strings — timezone-agnostic by construction.

Before — timezone-sensitive Date comparison

// new Date() parses a bare string as midnight UTC.
// A checkout/checkin on the same calendar day can
// appear to overlap once the property's UTC offset
// is applied.
function datesOverlap(checkOut, checkIn) {
  return new Date(checkOut) >= new Date(checkIn);
}

After — timezone-agnostic string comparison

// Compare calendar dates as YYYY-MM-DD strings.
// No Date object, no UTC offset, no false positives.
function datesOverlap(checkOut, checkIn) {
  const toDateString = (d) => new Date(d).toISOString().slice(0, 10);
  return toDateString(checkOut) >= toDateString(checkIn);
}

It's a small fix, but it's a reminder that a lot of production bugs are mismatch between what a data type represents and what you assume it represents. That's as true of a payout calculation as it is of a checkout date.

Lessons Learned

Synthetic monitoring across several live deployments caught issues because the failures depended on real-world timing. Soft-assertion patterns (collecting every conflict across every room type before raising a single alert) turned out to matter more here than in most software projects I've tested, because the cost of missing one instance is a guest turned away or double-booked.

I also learned how much operational trust depends on monitoring being predictable and continuous. A test suite that runs quietly in the background, on a regular automated schedule, and only makes noise when something is wrong, is worth more than one that runs once and is trusted forever.

Business Impact

The booking flow now runs with continuous, automated verification instead of manual checking. Guest enquiries get an immediate, accurate response regardless of time of day.

More than any single feature, building Kajuju's systems confirmed that hospitality software is fundamentally a coordination problem between inventory, staff, payments, and guest communication. Most failures live in the gaps between systems, not inside any one of them.
